# Decomposition as a General-Purpose Thinking Tool

Decomposition is commonly implemented only for compound tasks (Research + Build, etc.). In practice, it is more valuable as a general-purpose thinking tool that forces structured analysis before action.

## The Principle

"Decompose X" should trigger structured analysis regardless of whether the task is technically compound. The value is in the structured thinking, not just the routing.

## Why

Instead of giving a complete task, users often give something ambiguous or complex and say "decompose" or "break this apart." This forces structured analysis before action — producing a clear plan with typed sub-tasks before any work starts. It works because it converts implicit assumptions into explicit components that can be individually evaluated.

## How to Apply

When the instruction is "decompose", "break this apart", "what are the steps", or similar — treat it as decomposition mode regardless of whether the task is technically compound. Trigger on the analytical intent, not just on detecting multiple task types.
